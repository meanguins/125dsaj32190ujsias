Specificații Interactive & Ghid - Procesator Inteligent de CV-uri

Prezentare Generală a Sistemului

Această aplicație interactivă vă ghidează prin specificațiile tehnice, configurarea și utilizarea sistemului nostru avansat de procesare automată a CV-urilor. Sistemul este conceput ca o soluție robustă și scalabilă, menită să eficientizeze extragerea informațiilor structurate din CV-uri, utilizând cele mai noi tehnologii de inteligență artificială.

Arhitectura Sistemului și Tehnologii Cheie

Sistemul este construit ca o aplicație web FastAPI, containerizată cu Docker și optimizată pentru implementare pe platforme serverless precum Google Cloud Run.

  • API Backend: Dezvoltat în Python folosind framework-ul FastAPI.
  • Inteligență Artificială Generativă: Utilizează modelul `gemini-2.5-flash-preview-05-20` de la Google, accesat prin SDK-ul `google-genai`.
  • Bază de Date NoSQL: Google Cloud Firestore în modul Nativ.
  • Platformă de Rulare (Recomandată): Google Cloud Run.
  • Containerizare: Docker.

Funcționalități Detaliate

Sistemul nostru include o serie de funcționalități avansate pentru a asigura o procesare eficientă, sigură și fiabilă a CV-urilor. Fiecare componentă a fost proiectată cu atenție la detalii pentru a oferi o experiență optimă. Click pe fiecare titlu pentru a expanda detaliile.

Endpoint Webhook (`/webhook/cv`)

Metodă HTTP: POST

Securizare: Validare prin header `X-Webhook-Secret`.

Payload Intrare: CV (text/URL), `source_candidate_id`, `idempotency_key`, `metadata`.

Răspuns: HTTP 202 Accepted, `processing_id`, `status`, `status_url`.

Comportament: Validează, verifică idempotența, inițiază procesare asincronă.

Procesarea Fișierelor CV de la URL

Biblioteci: `aiohttp`, `PyPDF2`, `python-docx`.

Descărcare asincronă, User-Agent generic, detecție MIME.

Gestionare robustă a erorilor de rețea și parsare.

Integrarea cu API-ul Gemini

SDK: `google-genai`.

Model: `gemini-2.5-flash-preview-05-20`.

Configurație: `response_mime_type: "application/json"`, `response_schema`, `temperature: 0.1`, `safety_settings: BLOCK_NONE`, `thinking_config: thinking_budget=0`.

Prompt în engleză, apel asincron, reîncercări automate.

Persistența Datelor în Firestore

SDK: `firebase-admin`.

Structura Colecției: `artifacts/{APP_ID}/users/cv_processing_service_user/processed_cvs`.

Stochează: `processing_id`, payload, status, date extrase, erori, timestamps, etc.

Endpoint de Status

Metodă HTTP: GET, `/status/cv/{processing_id}`.

Returnează starea curentă și rezultatele unei procesări.

Gestionarea Idempotenței

Previne procesările duplicate bazate pe `idempotency_key` sau `source_candidate_id`.

Schema de Extracție a Datelor (`CV_EXTRACTION_SCHEMA`)

Aceasta este structura JSON detaliată pe care modelul Gemini o utilizează pentru a extrage și organiza informațiile din CV-uri. Puteți da click pe fiecare cheie principală pentru a vedea sub-proprietățile și descrierile acestora. Câmpurile marcate cu "(obligatoriu)" în descrierea schemei din codul sursă sunt esențiale pentru structura de bază a datelor extrase.

Ghid de Configurare a Sistemului (Setup)

Urmați acești pași pentru a configura și rula local sistemul de procesare a CV-urilor. Acest ghid presupune că aveți deja cunoștințe de bază despre Python, terminal și Google Cloud Platform.

1Cerințe Preliminare

  • **Python 3.10+** instalat pe sistemul dumneavoastră.
  • **Pip** (managerul de pachete Python) actualizat.
  • **(Opțional, dar recomandat pentru implementare) Docker** instalat.
  • Un **cont Google Cloud Platform (GCP)** activ, cu facturarea configurată.
  • **(Recomandat) Google Cloud SDK (`gcloud` CLI)** instalat și configurat pe mașina locală.

2Obținerea Codului Sursă

Descărcați sau clonați fișierele proiectului (`main.py`, `requirements.txt`, `Dockerfile`, etc.) într-un director local.

3Mediu Virtual și Dependențe

Este recomandat să folosiți un mediu virtual Python pentru a izola dependențele proiectului.

3.1. Creați și Activați Mediul Virtual:

Navigați în directorul proiectului și rulați:

python -m venv venv # Pe Windows: venv\Scripts\activate # Pe macOS/Linux: source venv/bin/activate

3.2. Instalați Dependențele:

Cu mediul virtual activat, instalați pachetele necesare:

pip install -r requirements.txt

4Configurarea Proiectului Google Cloud

4.1. Creați sau Selectați un Proiect GCP:

Accesați Google Cloud Console și selectați un proiect existent sau creați unul nou. Notați **ID-ul Proiectului (Project ID)**.

4.2. Activați API-urile Necesare:

Pentru proiectul selectat, asigurați-vă că următoarele API-uri sunt activate:

  • **Cloud Firestore API:** Pentru baza de date.
  • **Vertex AI API:** Pentru accesul la modelele Gemini. (Nota: Gemini API poate fi listat și sub "Generative Language API" sau similar, în funcție de interfața consolei).

Puteți activa API-urile din secțiunea "APIs & Services" > "Library" a consolei GCP.

5Configurarea Firestore

Dacă nu ați făcut-o deja la activarea API-ului:

  • Navigați la secțiunea Firestore din consola GCP.
  • Creați o bază de date în **Mod Nativ (Native Mode)**.
  • Alegeți o **locație** pentru baza de date (ex: `europe-west1` sau `eur5`). Această alegere este permanentă.

6Cont de Serviciu și Cheie JSON (pentru Dezvoltare Locală)

Pentru ca aplicația să se autentifice la GCP de pe mașina locală, veți avea nevoie de o cheie de cont de serviciu.

  • În consola GCP, navigați la "IAM & Admin" > "Service Accounts".
  • Click pe "+ CREATE SERVICE ACCOUNT".
  • Dați un nume contului (ex: `cv-processor-dev-sa`).
  • Acordați următoarele roluri contului de serviciu:
    • `Cloud Datastore User` (oferă acces la Firestore în Mod Nativ)
    • `Vertex AI User` (oferă acces la modelele Gemini și alte resurse Vertex AI)
  • Finalizați crearea contului de serviciu.
  • Selectați contul nou creat, mergeți la tab-ul "KEYS", click pe "ADD KEY" > "Create new key".
  • Alegeți tipul **JSON** și click pe "CREATE". Un fișier JSON va fi descărcat.
  • Păstrați acest fișier JSON într-un loc sigur.

**Alternativă (ADC):** Dacă aveți `gcloud` CLI instalat, rulați `gcloud auth application-default login` și autentificați-vă. Acest lucru configurează Application Default Credentials (ADC), iar aplicația le poate folosi automat dacă variabila `GOOGLE_APPLICATION_CREDENTIALS` nu este setată.

7Cheia API Gemini

Obțineți o cheie API pentru a utiliza modelele Gemini.

  • Mergeți la Google AI Studio.
  • Conectați-vă cu contul Google și creați o nouă cheie API dacă nu aveți deja una pentru proiectul GCP selectat.

8Configurarea Variabilelor de Mediu (`.env`)

Creați un fișier numit `.env` în directorul rădăcină al proiectului (lângă `main.py`) și adăugați următoarele variabile, înlocuind valorile placeholder:

GEMINI_API_KEY="CHEIA_DVS_API_GEMINI_AICI" WEBHOOK_SECRET="UN_SECRET_PUTERNIC_PENTRU_WEBHOOK" GCP_PROJECT_ID="ID_UL_PROIECTULUI_DVS_GCP" # Decomentați și completați linia de mai jos DOAR dacă NU folosiți ADC # GOOGLE_APPLICATION_CREDENTIALS="/calea/completa/catre/fisierul_serviciu_cont.json"

**Notă:** SDK-ul `google-genai` va căuta implicit variabila de mediu `GOOGLE_API_KEY`. Scriptul `main.py` este configurat să folosească valoarea din `GEMINI_API_KEY` pentru `GOOGLE_API_KEY` dacă aceasta din urmă nu este setată direct.

9Pornirea Locală a Serverului

Cu mediul virtual activat și fișierul `.env` configurat, porniți serverul FastAPI:

python main.py

Serverul ar trebui să ruleze pe `http://localhost:8000`. Verificați logurile din terminal pentru a confirma inițializarea corectă a serviciilor (Firestore, Gemini).

Ghid de Utilizare a Sistemului

Odată ce sistemul backend este configurat și rulează, puteți interacționa cu el prin API-ul expus.

1Endpoint Webhook: Procesare CV (`POST /webhook/cv`)

Acesta este endpoint-ul principal pentru a trimite CV-uri spre procesare.

Securitate:

Toate cererile către acest endpoint trebuie să includă header-ul HTTP `X-Webhook-Secret` cu valoarea secretului configurată în fișierul `.env`.

Payload-ul Cererii (Request Body - JSON):

Corpul cererii trebuie să fie un obiect JSON cu următoarea structură (bazată pe modelul Pydantic `WebhookPayload`):

{ "cv_text": "Opțional: Textul integral al CV-ului...", "cv_file_payload": { "cv_url": "Opțional: URL către fișierul CV (ex: PDF, DOCX)", "file_mime_type": "Opțional: Tipul MIME al fișierului din URL" }, "source_candidate_id": "Opțional: ID-ul candidatului din sistemul sursă", "idempotency_key": "Opțional: Cheie unică pentru idempotență", "metadata": { "cheie_custom": "valoare_custom" } }

**Notă:** Trebuie furnizat fie `cv_text`, fie `cv_file_payload.cv_url`.

Exemple de Cereri `curl`:

Înlocuiți `YOUR_WEBHOOK_SECRET` cu valoarea reală.

a) Trimitere CV ca text:

curl -X POST "http://localhost:8000/webhook/cv" \ -H "Content-Type: application/json" \ -H "X-Webhook-Secret: YOUR_WEBHOOK_SECRET" \ -d '{ "cv_text": "Nume: Ion Popescu...", "source_candidate_id": "text-cand-001", "idempotency_key": "req-text-001" }'

b) Trimitere CV ca URL:

curl -X POST "http://localhost:8000/webhook/cv" \ -H "Content-Type: application/json" \ -H "X-Webhook-Secret: YOUR_WEBHOOK_SECRET" \ -d '{ "cv_file_payload": { "cv_url": "https://example.com/cale/catre/cv.pdf", "file_mime_type": "application/pdf" }, "source_candidate_id": "url-cand-002" }'

(Pentru CMD Windows, comanda `curl` trebuie să fie pe o singură linie, cu ghilimelele interne escapate: `-d "{...}"` devine `-d \"{...}\"`).

Răspuns Așteptat (HTTP 202 Accepted):

{ "message": "CV primit și procesarea a fost inițiată în fundal.", "processing_id": "uuid-generat-automat", "status": "pending", "status_url": "http://localhost:8000/status/cv/uuid-generat-automat" }

Dacă se detectează o cerere duplicat (prin `idempotency_key` sau `source_candidate_id`), răspunsul va indica acest lucru, iar statusul va fi `already_received_or_processing`.

2Endpoint de Verificare a Statusului (`GET /status/cv/{processing_id}`)

Folosiți `processing_id`-ul returnat de endpoint-ul webhook pentru a interoga starea procesării.

Exemplu de Cerere `curl`:

Înlocuiți `ID_PROCESARE_PRIMIT` cu valoarea reală.

curl -X GET "http://localhost:8000/status/cv/ID_PROCESARE_PRIMIT"

Răspuns Așteptat (HTTP 200 OK):

Structura răspunsului depinde de starea procesării:

a) Procesare finalizată cu succes:

{ "processing_id": "ID_PROCESARE_PRIMIT", "status": "completed", "received_at_utc": "timestamp...", "updated_at_utc": "timestamp...", "extracted_data": { /* Datele extrase conform CV_EXTRACTION_SCHEMA */ }, "error_message": null, "processing_time_seconds": 15.7 }

b) Procesare în curs:

{ "processing_id": "ID_PROCESARE_PRIMIT", "status": "pending_processing", // sau "downloading_parsing", etc. "received_at_utc": "timestamp...", "updated_at_utc": "timestamp...", "extracted_data": null, "error_message": null, "processing_time_seconds": null }

c) Eroare în timpul procesării:

{ "processing_id": "ID_PROCESARE_PRIMIT", "status": "error", "received_at_utc": "timestamp...", "updated_at_utc": "timestamp...", "extracted_data": null, "error_message": "Descrierea erorii...", "processing_time_seconds": 5.2 }

Dacă `processing_id` nu este găsit, se va returna un răspuns HTTP 404 Not Found.

3Interpretarea Datelor Extrase (`extracted_data`)

Câmpul `extracted_data` din răspunsul de status (când procesarea este `completed`) va conține un obiect JSON structurat conform `CV_EXTRACTION_SCHEMA`. Puteți explora detaliile acestei scheme în tab-ul "Schema de Extracție" din această aplicație.

4Idempotența

Pentru a preveni procesarea accidentală a aceluiași CV de mai multe ori (și costurile asociate), sistemul suportă idempotența:

  • Furnizați un `idempotency_key` unic în payload-ul webhook-ului. Dacă o cerere cu aceeași cheie este primită din nou, sistemul va returna detaliile procesării inițiale fără a re-procesa CV-ul.
  • Ca alternativă, dacă `idempotency_key` nu este furnizat, dar `source_candidate_id` este, sistemul va folosi `source_candidate_id` pentru verificarea idempotenței.

Considerații Tehnice Suplimentare

Pentru o implementare robustă în producție, sunt luate în considerare următoarele aspecte:

Securitate

  • Utilizarea variabilelor de mediu și a Google Secret Manager pentru secrete.
  • Validarea riguroasă a input-urilor.

Scalabilitate

  • Arhitectură serverless (Cloud Run) pentru scalare automată.
  • Procesare asincronă pentru gestionarea eficientă a cererilor.

Robustețe

  • Mecanisme de reîncercare pentru apelurile API externe.
  • Gestionarea explicită a excepțiilor.

Logging și Monitorizare

  • Logging structurat pentru analiză în Cloud Logging.
  • Monitorizarea performanței și a erorilor.
